### The LibAFL Fuzzing Library
#### crates
- LibAFL is composed of different crates which is an individual library.
- `crates` list:
    - `libafl`: main crate, contains all the components needed to build a fuzzer.
        - choose features flags, some are already set by default, to configure the library:
            - `std`: enables the parts of the code that use the Rust standard library, only disables if you intent to use LibAFL in embedded environments.
            - `llmp_bind_public`: bind to a public TCP port, allows other fuzzers node can communicate with this instance.
            - `introspection`: adds performance statistics to LibAFL
    - `libafl_bolts`: contains useful low-level rust features.
        - `core_affinity`: bind curent process to cores.
    - `libafl_sugar`: `LibAFL`'s API, easy-to-use but not as flexible as stitching together from each individual component.
    - `libafl_derive`: exposes macro that can be used to define Metadata structs.
    - `libafl_targets`: instrument targets.
        - supported flags:
            - `pcguard_edges`: track the executed edges in a map.
            - `pcguard_hitcounts`: track the executed edges with the hitcounts (like AFL) in a map.
    - `libafl_cc`: provides utils to wrap compilers and create source-level fuzzers, currently, only Clang compiler is supported.
    - `libafl_frida`: bridges LibAFL with Frida as instrumentation backend, then instrument targets on Linux/macOS/Windows/Android for coverage collection.
    - `libafl_qemu`: bridges LibAFL with QEMU user-mode to fuzz ELF cross-platform binaries.
    - `libafl_nyx`: add capabilities of KVM-based snapshot fuzzer to LibAFL.
    - `libafl_concolic`: add symbolic execution engine to reach greater depth than normal fuzzing.

#### core concepts
- `observer`:
    - provides information observed during the execution of the program under test to the fuzzer.
    - the information is not preserved accross executions but it may be serialized and passed on to other nodes if an `Input` is considered `interesting`, and added to the `Corpus`.
- `executor`:
    - defines how to execute the target and all the volatile operations that are related to just a single run of the target, for instance, writing to a memory location or pass input as a parameter to the harness function.
    - can hold a set of `observer` connected with each execution, to obtain more information.
    - `InProcessExecutor`:
        - target is a harness function providing in-process crash detection.
        - pros: execute the harness as fast as possible.
        - cons: have to use another allocator if the harness is likely to have heap. corruption bug (because the corrupted heap maybe affect the fuzzer itself).
        => harness function là gì??? phải viết thêm nội dung gì đó để gọi target à???
        => `harness function` là một hàm phải tự viết trong fuzzer để bọc bên ngoài target. nó đóng vai trò như một bộ chuyển đổi dữ liệu thô từ fuzzer sang dữ liệu phù hợp mà target có thể hiểu được (ảnh, pdf, video,...) và gọi trực tiếp vào hàm API mục tiêu.
    - `ForkserverExecutor`:
        - implements an AFL-like mechanism to spawn child process to fuzz.
        - make a shared memory region to records coverage and pass this region to `observer` to optain coverage feedbacks combined with any `feedback`.
        - can improve performance by passing the input with shared memory too.
    - `InprocessForkExecutor`:
        - forks before running the harness to ensure the ubstable harness does't break things.
- `feedback`:
    - classifies the outcome of an execution of the program under test (the information reported by one or more `observers`) as interesting or not, if it is interesting, the corresponding input used to feed the target program is added to a corpus or maybe the result.
    - most of the time, the notion of `feedback` is deeply linked to `observer`.
    - the concept "interesting" related to a novelty search (compare with the past information).
- `input`:
    - the input of a program is the data taken from external sources that affect the program behavior.
    - the input of the program in the form of a byte array but the fuzzer doesn't represent the input in the same way that the program consumes it.
    - just before the execution, the data structure (for example, an abstract syntax tree,...) have to be serialized to a sequence of bytes.
    - an Input is a trait that can be implemented only by structures that are serializable (which means input can be converted to a sequence of byte) and have only owned data as field (it can't be reference to protect memory).
- `corpus`:
    - stores testcases on disk, memory or cache,...when it is considered interesting or fulfill an objective (crash the target,...).
    - the taxonomy of retrieving from the `corpus` is `scheduler`.
- `mutator`:
    - takes one or more input and generates a new instance of input.
    - can be compose and are generally linked to a specific input type.
    - this is a trait in LibAFL code.
- `generator`:
    - designed to generate input from scratch.
    - less used in feedback-driven fuzzing but there are some exceptions like Nautilus.
    - this is a trait in LibAFL code.
- `stage`:
    - an entity that operates on a single input received from the corpus.
    - there are several stages after retrieving input from the corpus, solves 2 main problems: mutation stage (mutate the same input multiple times) and analysis stage (trimming input or colorize important parts in input).
    - this is a trait in LibAFL code.
